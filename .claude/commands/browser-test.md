# 浏览器自测与 UI 视觉修复

在 Chrome 中测试网页并修复视觉问题。需要先通过 `claude --chrome` 或在会话中运行 `/chrome` 启用浏览器集成。

## 参数说明

| 参数 | 用途 |
|------|------|
| `visual` | 设计稿还原度检查 — 对比 Figma/设计稿描述，检查间距、颜色、字体、圆角等 |
| `form` | 表单验证测试 — 测试空值、边界值、非法输入、提交成功等场景 |
| `console` | 控制台错误调试 — 检查 JS 报错、网络请求失败、警告信息 |
| `responsive` | 响应式布局检查 — 在手机(375px)、平板(768px)、桌面(1440px)视口下检查布局 |
| `record` | 录制操作为 GIF — 记录用户操作流程，保存为可分享的 GIF 文件 |
| `flow` | 用户流程测试 — 模拟完整用户操作路径，检查每步是否正确渲染 |

如果不指定参数，将进入交互模式，逐个确认要执行的测试场景。

## 工作流程

### 1. 准备阶段

- 确认目标 URL（本地 `localhost:PORT` 或线上地址）
- 确认 Chrome 扩展已连接：运行 `/chrome` 检查状态
- 如果测试本地应用，确认开发服务器已启动

### 2. 执行测试

按照选定场景执行对应流程（见下方各场景说明），记录所有发现的问题。

### 3. 问题分析与修复

对于发现的每个问题：

1. **定位源码** — 找到问题对应的组件文件和样式代码
2. **分析原因** — 确定是样式错误、逻辑错误还是遗漏
3. **修复代码** — 直接在源码中修改
4. **回浏览器验证** — 刷新页面确认修复生效
5. **记录** — 将严重问题写入 `BUG_REPORT.md`

### 4. 输出报告

测试完成后输出结构化报告：

- 测试的页面和场景
- 发现的问题数量（按严重程度：严重/一般/建议）
- 已修复的问题列表
- 未能自动修复需要人工处理的问题
- 截图/GIF 存放路径

## 各场景详细流程

### visual — 设计稿还原度检查

```
目标：检查页面视觉是否与设计稿一致

检查清单：
□ 整体布局结构（header/sidebar/main/content/footer 是否位置正确）
□ 间距（margin/padding/gap 是否符合设计系统）
□ 颜色（背景色、文字色、边框色、hover/focus 状态色）
□ 字体（字号、字重、行高、字体族）
□ 圆角与阴影
□ 图标大小与对齐
□ 暗色模式（如适用）

操作步骤：
1. 打开目标页面
2. 读取页面 DOM 和 computed styles
3. 与设计稿描述/设计系统规范对比
4. 逐个元素列出差异
5. 修复 CSS/组件代码
6. 刷新页面验证
```

### form — 表单验证测试

```
目标：测试表单的验证逻辑和错误提示

测试矩阵：
- 空值提交：所有必填字段为空，检查错误提示
- 格式验证：邮箱格式错误、手机号位数不对、密码不符合规则
- 边界值：最大/最小长度、特殊字符、emoji、SQL 注入字符
- 正常提交：合法数据能否成功提交
- 网络异常：提交过程中断网，检查错误处理和重试

操作步骤：
1. 打开包含表单的页面
2. 按测试矩阵逐个场景操作
3. 检查错误提示文案和样式
4. 检查表单提交后的跳转/状态变化
5. 发现问题后修复代码并回归
```

### console — 控制台错误调试

```
目标：排查页面加载和运行时的 JS 错误

检查内容：
- console.error 和 console.warn
- 未捕获的 Promise rejection
- 网络请求 4xx/5xx 错误
- 资源加载失败（JS/CSS/图片 404）
- React/Vue 组件渲染警告

操作步骤：
1. 打开目标页面
2. 等待页面完全加载
3. 读取控制台输出（关注 error 和 warn 级别）
4. 点击页面关键交互元素后再次检查控制台
5. 根据报错定位源码并修复
6. 刷新验证
```

### responsive — 响应式布局检查

```
目标：确保页面在不同设备尺寸下正常显示

测试断点：
- 375px  (iPhone SE / 小型手机)
- 390px  (iPhone 14 / 常规手机)
- 768px  (iPad Mini / 平板竖屏)
- 1024px (iPad / 平板横屏)
- 1440px (标准桌面)
- 1920px (大屏桌面)

检查要点：
- 内容不超出屏幕边界（无横向滚动）
- 导航栏在移动端是否折叠为汉堡菜单
- 图片/视频是否等比缩放
- 表格/网格是否正确重排
- 字体大小在各断点是否可读
- 按钮和可点击区域是否大于 44px（移动端）

操作步骤：
1. 在每个断点宽度下打开页面
2. 截取全页截图（或描述布局情况）
3. 对比设计稿的响应式规范
4. 修复 CSS media query 或组件逻辑
5. 逐个断点验证
```

### record — 录制操作 GIF

```
目标：录制浏览器操作流程，保存为 GIF

操作步骤：
1. 确认起始页面 URL
2. 描述要录制的操作流程
3. Claude 执行操作并录制为 GIF
4. GIF 保存到项目根目录或指定路径

示例：
"录制一个 GIF 展示从首页搜索商品到完成下单的完整流程"
```

### flow — 用户流程测试

```
目标：模拟真实用户操作路径，端到端验证功能

操作步骤：
1. 定义用户流程的起点和终点
2. 逐步执行操作：点击、输入、选择、滚动
3. 每一步检查：
   - 页面是否正常加载
   - 操作响应是否符合预期
   - 有无视觉抖动/闪烁
   - 加载状态是否友好
4. 记录流程中的卡顿点和异常
5. 修复问题代码
```

## 快捷用法

```bash
# 快速检查当前开发页面
/browser-test visual  # 打开 localhost:3000 检查视觉还原度

# 测试表单
/browser-test form    # 测试页面中的表单验证

# 排查报错
/browser-test console # 检查控制台错误

# 全量测试（依次执行所有场景）
/browser-test all     # visual + form + console + responsive + flow
```
