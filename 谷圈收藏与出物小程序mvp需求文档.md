# 谷圈收藏与出物小程序 MVP 需求文档

## 1. 产品概述

### 1.1 产品名称

暂定名：**出物小助手**

备选名称：

- 谷子清单助手
- 吃谷记录本
- 谷圈出物助手
- 推し收藏册

### 1.2 产品定位

一款面向谷圈、小卡、娃圈、潮玩等收藏用户的微信小程序，帮助用户记录收藏、整理心愿单、生成出物清单图和多平台出物文案。

### 1.3 一句话说明

**帮谷圈/小卡/娃圈用户快速记录收藏，并一键生成可分享的出物长图和交易文案。**

### 1.4 目标用户

#### C 端用户

- 中重度吃谷用户
- 小卡收藏用户
- 棉花娃娃/BJD 玩家
- 退坑或阶段性出物用户
- 有收藏整理需求的二次元用户

#### 后续 B 端用户

- 拼团团长
- 代购团长
- 娃衣/手作店主
- 同人制品主理人
- 小卡团长

本期 MVP 主要面向 C 端用户，预留后续团长/店主版扩展。

---

## 2. 需求背景

### 2.1 用户痛点

1. 收藏品越来越多，难以记住自己买过什么。
2. 谷子、小卡、娃、潮玩等常常有预售、尾款、待发货、到货等状态，人工记录混乱。
3. 退坑或整理出物时，需要重复编辑图片、价格、瑕疵说明和交易规则。
4. 用户在闲鱼、小红书、微博、微信群等平台发布出物信息时，需要不同格式的文案。
5. 用户希望出物清单图更整洁、美观、可信。
6. 当前替代方案多为相册、备忘录、Excel、Canva、闲鱼草稿、小红书手动排版，流程割裂。

### 2.2 产品机会

本产品不做交易平台，不介入付款、担保、物流、售后纠纷，而是做轻量工具：

- 记录收藏
- 管理状态
- 生成出物清单
- 生成多平台文案
- 分享出物页

## 3. MVP 范围

### 3.1 本期必须做

| 模块          | 是否必须 | 说明                               |
| ------------- | -------- | ---------------------------------- |
| 微信登录      | 必须     | 获取用户身份，用于保存收藏数据     |
| 首页          | 必须     | 展示核心入口和最近数据             |
| 我的收藏      | 必须     | 收藏品列表、筛选、搜索             |
| 新增/编辑收藏 | 必须     | 录入图片、名称、角色、价格、状态等 |
| 出物清单生成  | 必须     | 勾选收藏品，生成清单图和文案       |
| 文案生成      | 必须     | 支持闲鱼、小红书、微博、微信群文案 |
| 分享页        | 必须     | 可分享出物清单给他人查看           |
| 心愿单        | 建议做   | 轻量记录想买的谷子/小卡/娃         |
| 模板选择      | 建议做   | 先做 2-3 个基础模板                |
| 水印          | 建议做   | 免费版生成图带小水印               |

### 3.2 本期不做

| 功能         | 不做原因                         |
| ------------ | -------------------------------- |
| 平台交易     | 涉及支付、退款、纠纷、风控，过重 |
| 担保交易     | 前期运营和合规成本高             |
| 陌生人聊天   | 审核和风控压力大                 |
| 社区广场     | 冷启动难，管理成本高             |
| IP 商品售卖  | 版权风险高                       |
| 真伪鉴定     | 需要专业能力，容易产生责任风险   |
| 行情估价     | 数据源难，不适合 MVP             |
| AI 角色陪聊  | 偏离核心工具定位                 |
| 复杂会员系统 | 可后续迭代                       |

# 5.4 新增/编辑收藏页

## 页面目标

让用户快速录入一件收藏品。

## 页面显示内容

### 图片区域

- 上传图片按钮
- 支持上传 1-9 张图片
- 第一张作为封面
- 支持拖动排序
- 支持删除图片

### 基础信息

字段如下：

| 字段     | 类型              | 是否必填 | 说明                         |
| -------- | ----------------- | -------- | ---------------------------- |
| 名称     | 文本输入          | 必填     | 商品名称，如“某某吧唧 A 柄”  |
| 作品     | 文本输入/历史选择 | 非必填   | 所属作品                     |
| 角色     | 文本输入/历史选择 | 非必填   | 所属角色                     |
| 类型     | 下拉选择          | 必填     | 吧唧、立牌、小卡等           |
| 入手价   | 数字输入          | 非必填   | 单位：元                     |
| 数量     | 数字输入          | 默认 1   | 支持多件                     |
| 入手渠道 | 文本/下拉         | 非必填   | 闲鱼、官店、拼团、代购等     |
| 入手日期 | 日期选择          | 非必填   | 默认当天                     |
| 当前状态 | 单选              | 必填     | 已到货、预售、待补款等       |
| 备注     | 多行文本          | 非必填   | 可记录瑕疵、柄图、捆绑规则等 |

### 状态选项

- 已到货
- 预售
- 待补款
- 待发货
- 待收货
- 待出物
- 已出物
- 不出

### 出物信息折叠区

默认收起，用户可展开。

字段：

| 字段         | 类型     | 说明                 |
| ------------ | -------- | -------------------- |
| 是否准备出物 | 开关     | 开启后可填写出物信息 |
| 出物价       | 数字输入 | 单位：元             |
| 瑕疵说明     | 多行文本 | 如有瑕疵需说明       |
| 是否包邮     | 单选     | 包邮/不包邮/满额包邮 |
| 是否可小刀   | 单选     | 可小刀/不刀          |
| 捆绑规则     | 文本     | 如“需捆 1 个普谷”    |

### 底部按钮

- 保存
- 保存并继续添加

## 页面交互

### 上传图片

- 点击上传图片，调用微信图片选择。
- 上传中显示 loading。
- 上传失败提示：图片上传失败，请重试。
- 超过 9 张提示：最多上传 9 张图片。

### 类型选择

- 点击后弹出底部选择器。
- 用户可选择系统类型或添加自定义类型。

### 状态选择

- 点击状态后弹出状态列表。
- 选择“预售”“待补款”“待发货”时，可提示用户设置提醒日期。

### 保存

保存前校验：

- 名称不能为空。
- 类型不能为空。
- 图片建议上传，但不强制。
- 入手价若填写，必须为大于等于 0 的数字。

保存成功后：

- 新增模式：返回收藏详情页。
- 编辑模式：返回上一页并刷新数据。

### 保存并继续添加

- 保存当前收藏。
- 清空表单。
- 保留上一次的作品、角色、类型，方便连续录入同系列收藏。

## 6. 核心业务流程

# 6.1 收藏录入流程

```text
首页/收藏页
→ 点击“记一件谷子”
→ 上传图片
→ 填写名称、类型、角色、价格、状态
→ 保存
→ 进入收藏详情页
```

# 6.2 生成出物清单流程

```text
首页/出物页
→ 点击“生成出物清单”
→ 选择收藏品
→ 填写出物价、瑕疵、包邮、捆绑规则
→ 选择模板
→ 生成长图
→ 保存图片 / 复制文案 / 分享
```

# 6.3 心愿转收藏流程

```text
心愿单
→ 点击某条心愿
→ 标记已入
→ 确认转为收藏
→ 自动带入信息到新增收藏页
→ 补充价格、状态
→ 保存收藏
```

# 6.4 分享转化流程

```text
用户 A 生成出物清单
→ 分享给用户 B
→ 用户 B 打开分享页
→ 查看清单
→ 点击“我也要生成出物清单”
→ 进入登录/首页
```

| 登录     | 用户可正常微信登录并保存数据                |
| 新增收藏 | 用户可新增含图片的收藏记录                  |
| 编辑收藏 | 用户可修改收藏信息                          |
| 收藏列表 | 支持查看、搜索、筛选                        |
| 出物清单 | 可选择多个收藏生成清单                      |
| 长图生成 | 至少支持 1 个稳定可用模板                   |
| 文案生成 | 至少支持闲鱼、小红书、微博、微信群 4 类文案 |
| 保存图片 | 可保存生成图到相册                          |
| 分享页   | 他人可通过链接查看清单                      |
| 异常提示 | 关键异常有明确提示                          |

### 13.3 性能验收

- 首页首屏加载时间尽量控制在 2 秒内。
- 收藏列表 100 条以内滚动流畅。
- 单次长图生成时间尽量控制在 5 秒内。
- 图片上传失败后可重试。

## 14. 风险与规避

### 14.1 版权风险

风险：用户上传的图片可能涉及 IP 内容。

规避：

- 产品仅作为个人记录和清单生成工具。
- 不主动售卖 IP 周边。
- 不使用未经授权的官方素材做商业宣传。
- 分享页增加免责声明。

### 14.2 交易纠纷风险

风险：用户通过分享页产生交易纠纷。

规避：

- 不介入交易。
- 不提供担保。
- 分享页提示交易需自行确认。
- 不展示“官方认证”“保真”等表述。

### 14.3 审核风险

风险：小程序被认定为交易平台或社区平台。

规避：

- 初期不做支付闭环。
- 不做商品广场。
- 不做陌生人私聊。
- 定位为图片工具、收藏管理、效率工具。

### 14.4 用户冷启动风险

风险：用户不愿意手动录入收藏。

规避：

- 支持连续添加。
- 支持复制上一条信息。
- 支持从出物清单反向创建收藏。
- 后续支持批量导入。

---

## 15. 产品总结

MVP 的核心不是做一个完整二次元平台，而是先验证一个具体场景：

**用户是否愿意在小程序里记录收藏，并用它生成出物清单。**

第一版的关键体验必须做到：

1. 录入足够快。
2. 清单生成足够简单。
3. 长图足够好看。
4. 文案能直接复制使用。
5. 分享页能带来新用户。

若这些路径跑通，再向收藏管理、模板付费、会员、团长排单工具迭代。

## 16.1 原型设计总原则

### 设计风格

- 风格关键词：清爽、可爱、轻量、二次元友好、不幼稚。

# 17. 前后端接口文档

本章节定义小程序前端与后端服务之间的接口。接口风格采用 RESTful JSON API，实际开发时可根据云开发、Serverless、Node.js、Java、Go 等技术栈调整。

---

## 17.1 接口通用约定

### 基础路径

```text
/api/v1
```

### 请求格式

除文件上传接口外，请求体均使用 JSON：

```http
Content-Type: application/json
```

### 响应格式

统一响应结构：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

### 分页响应结构

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "list": [],
    "pagination": {
      "page": 1,
      "page_size": 20,
      "total": 100,
      "has_more": true
    }
  }
}
```

### 通用错误码

| code | 含义              | 说明                |
| ---- | ----------------- | ------------------- |
| 0    | success           | 成功                |
| 400  | bad_request       | 请求参数错误        |
| 401  | unauthorized      | 未登录或 token 失效 |
| 403  | forbidden         | 无权限              |
| 404  | not_found         | 资源不存在          |
| 409  | conflict          | 数据冲突            |
| 413  | payload_too_large | 文件或请求体过大    |
| 429  | too_many_requests | 请求过于频繁        |
| 500  | server_error      | 服务端错误          |

### 鉴权方式

登录成功后后端返回 token。前端后续请求在 Header 中携带：

```http
Authorization: Bearer <token>
```

### 时间格式

统一使用 ISO 8601 字符串：

```text
2026-05-19T10:30:00+08:00
```

### 金额格式

金额统一使用分或小数元均可。MVP 建议前后端使用“元”的 number 类型，展示时保留 2 位小数。

```json
{
  "purchase_price": 25.5
}
```

## 17.13 前端状态管理建议

### 需要缓存的数据

- 用户信息
- 首页统计
- 收藏类型枚举
- 模板列表
- 最近搜索关键词
- 出物清单草稿

### 本地草稿

新增收藏和编辑出物信息建议支持本地草稿，避免用户中途退出丢失数据。

本地草稿场景：

- 新增收藏未保存
- 出物清单编辑中
- 反馈内容未提交

### 图片上传策略

建议图片选择后立即上传，后端返回 URL，表单保存时仅提交 URL。

好处：

- 表单提交更快。
- 失败可单独重试。
- 长图生成时可直接使用远程图片。

## 17.14 后端实现建议

### 服务模块

建议后端拆分为：

1. AuthService：登录鉴权
2. UserService：用户信息
3. UploadService：图片上传
4. CollectionService：收藏管理
5. SaleListService：出物清单管理
6. GenerateService：长图和文案生成
7. ShareService：分享页
8. WishlistService：心愿单
9. TemplateService：模板
10. FeedbackService：反馈
11. EventService：埋点

### 数据库表

建议至少包含：

- users
- collection_items
- sale_lists
- sale_list_items
- wishlist_items
- templates
- shares
- feedbacks
- event_logs

### 图片存储

- 使用云对象存储。
- 存储路径按用户和场景区分。

示例：

```text
/images/{user_id}/collection/{date}/{file_id}.jpg
/images/{user_id}/generated/{sale_list_id}.jpg
```

### 长图生成

可选方案：

1. 前端 Canvas 生成
2. 后端使用 Canvas/Puppeteer 生成
3. 云函数生成

MVP 建议：

- 如果模板简单，可前端 Canvas 生成，降低后端复杂度。
- 如果需要统一水印、高清导出、复杂模板，后端生成更稳定。

### 文案生成

MVP 阶段不一定需要大模型。可用规则模板生成：

```text
出物整理，共 {count} 件。
价格见图，{shipping_rule}。
{bargain_rule}，{bundle_rule}。
有瑕疵已标注，介意慎。
```

后续可增加 AI 优化文案。

## 17.17 接口验收标准

### 通用验收

- 所有接口返回统一响应结构。
- 所有需要登录的接口在 token 缺失时返回 401。
- 用户只能访问自己的收藏、心愿、出物清单。
- 分享页接口可匿名访问。
- 参数错误时返回明确错误信息。

### 收藏接口验收

- 可新增含 1-9 张图片的收藏。
- 可按状态、类型、关键词筛选。
- 可编辑所有字段。
- 删除后列表不再返回该收藏。

### 出物接口验收

- 可基于多个收藏创建清单。
- 可编辑清单商品价格和规则。
- 可选择模板生成图片。
- 可返回至少 4 类平台文案。
- 生成后可通过 share_id 访问分享页。

### 分享接口验收

- 未登录用户可打开分享页。
- 分享页不暴露发布者隐私信息。
- 分享页显示免责声明。
- 点击“我也要生成”可进入登录或首页。

# 18. 技术选型与基础架构

## 18.1 前端

- 框架：微信小程序原生 + TypeScript（首选）；如后期要复用 H5，可平滑迁移到 Taro 3。
- 状态管理：MobX-miniprogram。
- 网络层：统一封装 `wx.request`，自动处理 401（access token 过期 → 刷新）、loading、重试（最多 1 次，仅 GET）。
- 列表：`scroll-view` + 触底加载；超过 200 条启用虚拟列表。
- 长图渲染：`canvas type="2d"` + `wx.canvasToTempFilePath`，作为后端生成失败时的兜底方案。
- 包体策略：图片资源全部走 CDN，主包不超过 1.5 MB。

## 18.2 后端

- 语言/框架：Node.js 20 + NestJS + TypeScript。
- ORM：Prisma。
- 数据库：MySQL 8（utf8mb4），主从读写分离（v0.3 启用）。
- 缓存：Redis 6（缓存、限流计数、refresh token、分享 PV 计数）。
- 对象存储：阿里云 OSS 或腾讯云 COS（接 CDN，私有 Bucket + 临时签名）。
- 任务队列：BullMQ（长图生成、内容审核回调）。
- 鉴权：JWT（access 7 天 + refresh 30 天）。

## 18.3 长图生成

- MVP 方案：**服务端 Puppeteer + Headless Chrome** 渲染 HTML/CSS 模板 → 截图 PNG。
  - 优点：模板可热更新（HTML/CSS 存 `templates.config`），样式可控。
  - 缺点：单实例并发有限，需要 Worker 池。
- 性能目标：单图渲染 ≤ 3 秒（不含上传 OSS）。
- 并发策略：Worker 池 4 进程；超过排队，前端 loading 提示"正在生成第 N/M 张"。
- 兜底：服务端连续失败 2 次，前端用 Canvas 自行生成（仅"简洁表格款"支持）。

## 18.4 部署架构

```text
小程序 → CDN → API 网关 → NestJS 服务（K8s, 2 副本）
                          ├─→ MySQL（主从）
                          ├─→ Redis
                          ├─→ OSS + CDN
                          └─→ Puppeteer Worker（独立 Pod）
```

## 18.5 兼容性

- 微信基础库 ≥ 2.30.0。
- 适配机型：iPhone SE（375×667）至 iPhone 15 Pro Max；Android 主流机型（小米、华为、OPPO、vivo）。
- iOS / Android 各自走一遍 P0 路径验收。
- PC 端微信不支持，登录页提示"请使用手机微信打开"。

# 21. 收藏 ↔ 出物清单状态同步规则

## 21.1 关系定义

- 一件 `CollectionItem` 可被加入 0 ~ N 个 `SaleList`。
- 加入清单时执行**字段快照**：`name`、`cover_image`、`sale_price → price`、`flaw_note`、`shipping_rule`、`bargain_rule`、`bundle_rule`、`note` 全部写入 `SaleListItem.collection_snapshot`。
- 快照后清单内字段独立，**不**随收藏后续编辑而变化（保护已生成长图与已分享清单的历史一致性）。

## 21.2 收藏状态机（CollectionItem.status）

```text
preorder → pending_payment → pending_shipment → pending_receipt → arrived
arrived  → for_sale → sold
arrived  → not_for_sale
* → 用户可手动跳转至任意状态（产品自由度）
```

- `is_for_sale` 由系统计算：当且仅当 `status = for_sale` 时 `is_for_sale = true`。

## 21.3 清单商品状态机（SaleListItem.status）

```text
available → reserved → sold
available → sold （直接出）
sold      → available（撤回）
```

## 21.4 联动规则表

| # | 触发动作 | 收藏侧变化 | 清单侧变化 |
|---|---|---|---|
| 1 | 收藏被加入清单 | `status` 改为 `for_sale`（若原为 `arrived`），`is_for_sale=true` | 新增行 `available`，落快照 |
| 2 | 清单中标记 `sold` | 该收藏 `status → sold`，`is_for_sale=false` | 当前行 `sold`，更新 `sold_at` |
| 3 | 收藏侧"标记已出" | `status → sold`，`is_for_sale=false` | **所有引用此收藏的清单行**自动 `sold`（事务内） |
| 4 | 编辑收藏字段 | 收藏更新 | 清单**不变**（保留快照） |
| 5 | 软删收藏 | `deleted_at` 写入 | 清单行保留快照，前端展示"原收藏已删除"灰色角标 |
| 6 | 软删清单 | `is_for_sale` 重新计算（若该收藏未被其他清单引用，恢复 `arrived`） | 清单及其行级联软删 |
| 7 | 撤回 sold（清单中改 available） | 若收藏未在其他清单 `sold` 中，`status` 恢复 `for_sale` | 当前行 `available` |
| 8 | 收藏从清单中移除 | `is_for_sale` 重新计算 | 删除该清单行（硬删，因有快照可追溯） |

## 21.5 一致性保证

- 上述跨表更新一律在数据库事务内执行；关键场景 #3 涉及多行更新，使用 `SELECT ... FOR UPDATE` 锁。
- 兜底任务：每日凌晨 03:00 跑一次 `recompute_is_for_sale`，校正可能的漂移。
- 接口幂等：所有"标记已出"、"撤回"、"移除"接口需支持重复调用幂等（带 `request_id`）。

## 21.6 前端同步策略

- 详情页/清单页打开时强制走最新接口，不读缓存。
- 列表页内修改后，本地乐观更新 + 接口确认；失败回滚并提示"操作失败，请刷新"。

# 22. 合规与内容安全

## 22.1 微信小程序合规

- 服务类目：**工具 / 效率工具**。
- **不**申请：电商、社区、社交（避免被认定为交易/社区平台）。
- 主体：建议个人或工作室主体即可（MVP 阶段）。
- ICP 备案：使用境内服务器需要 ICP 备案。
- 隐私接口列表（小程序后台需配置）：`wx.login`、`wx.getUserProfile`、`wx.chooseImage` / `wx.chooseMedia`、`wx.saveImageToPhotosAlbum`。

## 22.2 内容审核（**P0 必做**）

### 22.2.1 图片审核

- 上传成功后，后端异步调用微信 `security.imgSecCheck` 或腾讯云内容安全。
- 状态字段：`audit_status ∈ { pending, passed, rejected }`。
- 命中违规：`rejected` → 软删图片 + 通知用户（违规计数 +1）。
- 疑似命中：留 `pending`，列表/分享页**不展示**该图（用占位代替）；人工复核后改 `passed`。
- 单条收藏所有图片审核结果聚合：任一 rejected → 整条收藏 `audit_status=rejected`，列表不展示。

### 22.2.2 文本审核

- 文本字段：收藏 `name / note / flaw_note`、清单 `title / description`、心愿 `name / note`、反馈 `content`。
- 入库前调用 `security.msgSecCheck` 同步校验，命中违规直接拒绝（返回 `code: 6001`）。
- 异步抽检：每日凌晨对前一天数据做二次抽样审核。

### 22.2.3 违规处置

| 违规次数（90 天滑动窗） | 处置 |
|---|---|
| 1 次 | 内容拒绝 + 提示 |
| 2 次 | 内容拒绝 + 24 小时禁止上传/生成 |
| 3 次及以上 | 账号封禁 7 天 |
| 严重违规（涉黄/政/暴力） | 永久封禁 |

## 22.3 隐私与协议

- 收集信息清单（写入隐私政策）：
  - 必收：openid、登录态。
  - 选收：unionid、昵称、头像、用户上传图片、用户输入文本、设备机型、操作日志。

# 24. 鉴权、限流与安全策略

## 24.1 JWT Token 策略

- **access token**：HS256，有效期 7 天，载荷含 `user_id / openid / role / iat / exp`。
- **refresh token**：HS256，有效期 30 天，存 Redis（key: `refresh:{user_id}`），可被强制吊销。
- 双 token 流程：
  1. 接口返回 `code: 1002` → 前端拦截器自动调用 `POST /auth/refresh`。
  2. refresh 成功 → 重发原请求；失败（`1003`）→ 清空本地态，跳登录页。
- 主动登出：access token 写入黑名单（Redis，TTL = 剩余有效期）。
- 多端登录：允许，最多 3 个设备；超过 3 个时挤掉最早登录的设备。

## 24.2 新增接口：刷新 Token

```http
POST /api/v1/auth/refresh
```

请求：

```json
{
  "refresh_token": "xxx"
}
```

响应：

```json
{
  "code": 0,
  "data": {
    "token": "new_access_token",
    "refresh_token": "new_refresh_token",
    "expires_in": 604800
  }
}
```

> refresh token 一次性使用：每次刷新会签发新的 refresh token，旧的失效（防重放）。

## 24.3 限流策略

| 范围 | 阈值 | 实现层 |
|---|---|---|
| 全局每 IP | 1000 次/分钟 | 网关层（Nginx limit_req） |
| 单用户全部接口 | 60 次/分钟 | Redis incr + TTL |
| 长图生成 | 3 次/天（免费版） | 业务计数 |
| 上传图片 | 30 次/分钟 | Redis incr |
| 文案复制埋点 | 100 次/分钟 | Redis incr |
| 反馈提交 | 5 次/天 | 业务计数 |
| 注册 | 单 IP 5 个/小时 | Redis incr |

超限响应：HTTP 429 + `code: 9001/9002`，响应头携带 `Retry-After`。

## 24.4 图片访问安全

- OSS Bucket 权限：私有读写。
- 接口返回的图片 URL 为带签名的临时链接，有效期 2 小时。
- 长图（生成结果）链接有效期 7 天，可重新生成签名。
- CDN 防盗链白名单：仅小程序业务域名 + 微信分享域名。
- 上传时校验 MIME（前 8 字节魔术字识别），防伪装上传。

## 24.5 分享 ID 安全

- 格式：`base62(sha256(sale_list_id + salt + created_at)[:12])`，防顺序遍历。
- 撤回机制：用户在"我的分享"中可关闭分享，关闭后访问返回 `3006`（HTTP 410）。
- 单分享页 PV > 1 万 → 自动进入审核队列（人工复核内容是否违规）。
- 分享页**不**返回发布者的 `openid / 真实 ID`，仅返回 `nickname / avatar`（用户可在隐私设置中关闭头像/昵称展示）。

## 24.6 数据隔离

- 鉴权中间件强制注入 `user_id`，所有 ORM 查询都携带 `WHERE user_id = ?`。
- Service 层在执行写操作前二次校验 `record.user_id === ctx.user_id`，防止越权。
- 后台管理接口使用独立 `/admin/v1` 前缀和独立鉴权（管理员账号，**非**用户 token）。

## 24.7 接口签名（v0.3 启用）

- MVP 阶段不强制；
- v0.3 起对支付、敏感写接口加 HMAC-SHA256 签名，防中间人篡改。

- 角色：
  - `super_admin`：全部权限。
